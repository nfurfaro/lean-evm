# Sprint 5: Spec → EVM Bytecode (Shallow PoC)

**Date:** 2026-02-24
**Status:** Design
**Goal:** Generate Solidity-ABI-compatible EVM bytecode from the mint spec, deploy to anvil, verify with cast.

---

## Motivation

Sprints 1–4 built a verified spec layer: dependent types encode smart contract properties, a DSL lets developers write specs declaratively, and proofs are machine-checked. But there's no bridge to executable code. The spec proves *what* should happen; this sprint generates code that *does* it.

Two possible directions: spec → code (compilation) or code → spec (verification of existing code). We chose **spec → code** because:
- The spec is the source of truth — generated code is correct by construction
- No need to formalize Solidity's semantics (the harder problem)
- Shorter trust chain: Lean kernel → codegen → bytecode (no solc in the loop)

We target **EVM bytecode directly**, not Solidity source, because:
- solc is unverified — it has had bugs that changed program semantics
- Skipping solc eliminates an entire class of compiler bugs from the trust chain
- The generated bytecode doesn't need to be human-readable if the pipeline is verified

## Scope

**In scope:**
- EVM opcode model (Lean inductive type, ~17 opcodes)
- Assembler: `List Op → ByteArray` with label resolution
- Codegen for `mint(address, uint256)` — the spec's simplest atom
- Codegen for getters: `totalSupply()` and `balanceOf(address)`
- Function selector dispatch (3 functions)
- Deployment (constructor) bytecode wrapper
- `#eval` that outputs deployable hex
- End-to-end test: deploy to anvil, call mint, read back state

**Out of scope:**
- Burn, transfer, or any other atom
- Event emission (LOG opcodes)
- Access control (onlyOwner)
- Overflow checks
- Hooking into the `spec` macro
- Formal verification of the codegen

## Architecture

```
spec mint ...  (DSL — already exists)
      │
      ▼  (this sprint, manual wiring)
mintCode : List Op  (pure Lean function)
      │
      ▼
assemble : List Op → ByteArray
      │
      ▼
#eval → hex string
      │
      ▼
anvil + cast  (deploy & test)
```

The codegen is decoupled from the spec macro. This sprint builds it as standalone Lean definitions. A future sprint wires the macro to invoke codegen automatically.

### Design for future verification

The compiler will ultimately be formally verified. Even though this sprint doesn't prove correctness, the structure supports future proofs:

1. **Pure functions only.** Codegen is `Spec → List Op`, assembler is `List Op → ByteArray`. No `IO`, no `MetaM`. Pure functions are what Lean can write theorems about.

2. **Single source of truth for constants.** Opcode byte values, storage slots, ABI selectors — each defined once as a `def`, not scattered as magic numbers.

3. **Semantic types at boundaries.** Codegen works with `Op`, not `UInt8`. The assembler is the only `Op → bytes` boundary. This enables independent verification: prove the codegen produces the right `List Op`, then prove the assembler faithfully serializes.

4. **The correctness theorem (future):**
   > For any EVM state where slot 0 holds `s` and the mapping slot for `dst` holds `b`, executing `assemble (mintCode dst amt)` on calldata encoding `(dst, amt)` results in a state where slot 0 holds `s + amt` and the mapping slot holds `b + amt`.

   This decomposes into:
   - **Codegen correctness:** `mintCode` produces opcodes whose semantics match the spec's postconditions
   - **Assembler correctness:** `assemble` faithfully encodes `Op` to bytes

   Both require a formal EVM semantics for the opcode subset (`exec : State → Op → State`), which is future work. The opcode inductive type built here becomes its foundation.

## EVM Opcode Model

Opcodes needed for mint + getters + dispatch:

| Category | Opcodes |
|----------|---------|
| Stack | `PUSH1`, `PUSH4`, `DUP1`–`DUP3`, `SWAP1`, `POP` |
| Arithmetic | `ADD`, `SHR` |
| Comparison | `EQ` |
| Memory | `MSTORE`, `RETURN` |
| Storage | `SLOAD`, `SSTORE` |
| Calldata | `CALLDATALOAD` |
| Crypto | `SHA3` (keccak256 — mapping slot computation) |
| Control | `JUMP`, `JUMPI`, `JUMPDEST`, `REVERT`, `STOP` |
| Deploy | `CODECOPY` |

Modeled as:

```lean
inductive Op where
  | push1 (val : UInt8)
  | push4 (val : UInt32)
  | add | shr | eq
  | mstore | sload | sstore
  | calldataload | sha3
  | jump | jumpi | jumpdest
  | dup1 | dup2 | dup3 | swap1 | pop
  | revert | stop
  | ret       -- RETURN (0xf3)
  | codecopy
```

## Storage Layout (Solidity-compatible)

```
Slot 0: totalSupply (uint256)
Slot 1: balances mapping
  └─ balances[addr] stored at keccak256(abi.encode(addr, 1))
```

Mapping slot computation happens at EVM runtime via `SHA3` — Lean emits the instructions, doesn't compute the hash.

## ABI (Solidity-compatible)

| Function | Selector | Calldata |
|----------|----------|----------|
| `mint(address,uint256)` | `0x40c10f19` | `[4:36] = dst`, `[36:68] = amt` |
| `totalSupply()` | `0x18160ddd` | (none) |
| `balanceOf(address)` | `0x70a08231` | `[4:36] = addr` |

## Bytecode Structure

### Runtime bytecode

```
┌─────────────────────────────┐
│ Function selector dispatch  │  Extract selector, compare, jump
│   match mint      → JUMPI   │
│   match totalSupply → JUMPI │
│   match balanceOf → JUMPI   │
│   fallthrough     → REVERT  │
├─────────────────────────────┤
│ mint logic                  │  CALLDATALOAD args, update storage, STOP
├─────────────────────────────┤
│ totalSupply getter          │  SLOAD slot 0, MSTORE, RETURN
├─────────────────────────────┤
│ balanceOf getter            │  Compute mapping slot, SLOAD, MSTORE, RETURN
└─────────────────────────────┘
```

### Deployment bytecode

```
PUSH <runtime_size>
PUSH <runtime_offset>
PUSH 0x00
CODECOPY
PUSH <runtime_size>
PUSH 0x00
RETURN
<runtime bytecode>
```

## Kill Criteria

| Criterion | How to verify |
|-----------|---------------|
| `lake build` succeeds | Bytecode generated at compile time, no errors |
| Bytecode deploys to anvil | `cast send --create 0x<hex>` returns contract address |
| `mint(addr, 100)` updates state | `totalSupply()` returns 100, `balanceOf(addr)` returns 100 |
| Mint accumulates correctly | Two `mint` calls, values sum correctly |

## What This Proves

If the kill criteria pass, we've demonstrated that:
1. A Lean program can generate valid, deployable EVM bytecode
2. The generated bytecode correctly implements the mint spec's semantics (empirically, not formally)
3. The architecture supports future formal verification of the compiler itself
4. No Solidity compiler is needed — the trust chain is Lean → bytecode with no unverified intermediary
